Patterns

System Messages

Should be specific and inform through, polite, and precise natural language with constructive guidance on what to do next.

System messages are an important part of providing the user with visibility into how their account is performing. These messages might come in a variety of formats and styles appropriate to each scenario.

Input Validation

Use validation messages to indicate when a form or field submission fails, or requires additional information to be shown.

Input Validation Key #
  1. Label: Indicates what sort of information the field requires and should be short and concise. Labels are directly above their input and aligned to the left.
  2. Placeholder text: Let people know how the field should be used. Only add where additional clarification is needed.
  3. Input field: Area where the text will be entered.
  4. Helper text: Gives context about a field's input. It's best to provide an example or specific syntax for the input here, rather than in the placeholder, so that it's visible after text has been entered. Only use this where clarification is required, and try not to overuse it.
  5. Active input field: When the field is selected for text entry its border turns blue and a light blue spotlight is added to help focus.
  6. Input field validation: When an error related to the field occurs the border turns red to highlight the error.
  7. Validation error: Displays the error message related to the restrictions of the field. Keep this text as short as possible.
  8. Validation icon: Helps indicate what state the field has.
  9. Active input field validation: When the field with the error is selected for text entry a light red spotlight is added to help focus.

Alerts #

Use alerts at the top of the screen or section to display critical messaging about loss of data, functionality, or important site-wide information that affects the user's ability to use the product. Alerts generally should appear one at a time, and should only disappear once the related issue is resolved, and generally are not dismissible. Alerts appear in the screen by pushing the entire content below by animating.

Alert Key #
  1. Description: This should describe the issue and any action the user needs to take, if any. The alert will expand with the length of the description but should be as succinct as possible.
  2. Icon (optional): An icon can be added to help distiguish the type of alert see the variations in the chart below.
  3. Actions (optional): One or more links that enable the user to act on the messages content.
  4. Dismissable (optional): Gives the user the option to dismiss the alert; this should only be used with alerts that are informative.

Notifications #

Use toast notifications to communicate conditions, indicate an event, or to show responses to user actions. Content can include authentication, information, confirmation, warnings, and error messages. It is important to specify which of these the content of the message falls under in order to choose the correct type.

Toast notifications by default should appear on top of content in the top right of the viewport just below the page header.

Notification Key #
  1. Title: Toast notifications should always contain a concise title.
  2. Time Stamp (optional): Informs the user when the event occurred.
  3. Dismissable: Gives the user the option to dismiss the notification.
  4. Description: Descriptions should be concise and limited to three lines in length.

Full Page #

Use full page messages when a complicated process has been completed or fails and it doesn't make sense to leave the user on the current page.

These messages are presented on a card and top-centered on their own page.

Full Page Messaging Key
  1. Icon: The icon indicates the status of the message. The icon's color and symbol give a quick visual indicator of the type and urgency of the message.
  2. Title: A concise title should give the user context for the message.
  3. Description: Descriptions should be informative in HealthEquity's voice and limited to three lines in length.
  4. Button: A button that enable users to act or move on from the message can be presented.
  5. Links: A maximum of two links that enable users to act or move on from the message should be presented.
